# Policy RAG — Full Tech Stack Documentation

## Project Overview
A production-grade **Retrieval-Augmented Generation (RAG)** system that allows users to upload documents (PDF, DOCX, TXT) and media files (MP4, MP3, WAV) and ask natural language questions. Answers are generated by an LLM using only the content from the uploaded documents. Supports **multi-modal content**: text, tables, and audio/video transcripts. Includes **text-to-speech** playback of answers and **orphan index cleanup**.

## AI / ML Stack

| Component | Technology | Details |
|---|---|---|
| Embedding Model | `all-MiniLM-L6-v2` | SentenceTransformers — 384-dim vectors |
| Vector Database | ChromaDB | PersistentClient, local SQLite-backed |
| LLM | llama-3.1-8b-instant | Via Groq API (OpenAI-compatible) |
| LLM Protocol | OpenAI SDK | Streaming (`stream=True`) via SSE |
| Query Expansion | LLM-based | Fast non-streaming call, generates 2 alternative phrasings |
| Retrieval | Multi-Query Semantic | Run ChromaDB for original + 2 expanded queries, deduplicate by chunk_id |
| Reranking | BM25 + RRF | IDF-weighted keyword (BM25Okapi) + reciprocal rank fusion (k=60) |
| Context Window | Parent-child expansion | Fetch adjacent chunks (N-1, N, N+1) from same source, merge into 9 sentences |

### RAG Pipeline Flow (with Multi-Doc Context Window + Hybrid Search)
```
User Question
     │
     ▼
Query Expansion (LLM) → [original, alt1, alt2]
     │
     ▼
Multi-Query Retrieval (3× ChromaDB calls)
     ├─ Query embedding (all-MiniLM-L6-v2)
     ├─ Similarity search (initial_k = top_k × 6 = 48 candidates max)
     └─ Deduplicate across queries by chunk_id
     │
     ▼
Hybrid Reranking (BM25 + RRF)
     ├─ Semantic ranking: sort by L2 distance
     ├─ BM25 ranking: IDF-weighted term frequency (Okapi BM25)
     └─ RRF fusion: 1/(rank_sem + 60) + 1/(rank_bm25 + 60)
     │
     ▼
Sort by Document Order (group by source, sort by chunk index)
     │
     ▼
Context Window Expansion (fetch adjacent chunks N-1, N, N+1)
     │
     ▼
Top-K Chunks (default: 12, expanded to ~9 sentences each) → Build Prompt
     │
     ▼
Groq LLM API (streaming)
     │
     ▼
Token-by-token SSE stream → Browser
```

### Chunking Strategy
| Setting | Value | Reason |
|---|---|---|
| Sentences per chunk | 3 | Smaller = more focused embeddings |
| Overlap | 1 sentence | Preserve context between chunks |
| Batch upsert size | 100 chunks | Avoid SQLite variable limit |
| File check | Per-file `collection.get(where={"source": f})` | Avoid "too many SQL variables" error |

## Document Processing

| Format | Library | Method |
|---|---|---|
| PDF (text) | PyMuPDF (`fitz`) | Page-by-page text extraction |
| PDF (tables) | PyMuPDF (`fitz`) | `page.find_tables()` → markdown conversion |
| DOCX (text) | python-docx | Paragraph-by-paragraph extraction |
| DOCX (tables) | python-docx | `doc.tables` → markdown conversion |
| TXT | Built-in Python | 1MB block streaming |
| MP4/WebM | ffmpeg / moviepy | Audio extraction → Whisper transcription |
| MP3/WAV/M4A | Groq Whisper API | Direct transcription with timestamps |
| Chunking | NLTK `sent_tokenize` | Sentence-aware splitting |

### Multi-Modal Chunk Types
| Type | ID Format | Adjacent Expansion | Metadata |
|---|---|---|---|
| `text` | `{file}_chunk_{N}` | Yes (N-1, N, N+1) | `source`, `page` |
| `table` | `{file}_table_{page}_{idx}` | No (self-contained) | `source`, `page`, `table_index` |
| `transcript` | `{file}_transcript_{N}` | Yes | `source`, `timestamp` |

### Storage Architecture
```
/app/data/              ← Azure File Share (SMB mount) — files persist across deployments
/app/index/             ← Local container SSD — fast ChromaDB queries
    chroma_db/
        chroma.sqlite3

Azure Blob Storage      ← ChromaDB index backup
    policyrag-index/
        chroma_index/
            chroma.sqlite3
```

### Index Persistence Strategy
```
Container Startup:
  → Download chroma_index/ from Blob → /app/index/chroma_db (fast local)

After each Ingest:
  → Upload /app/index/chroma_db → chroma_index/ in Blob (background)

After each Delete:
  → Upload updated index to Blob (background thread)

Result: Index survives deployments without re-indexing all files
```

## Key Design Decisions

| Decision | Reason |
|---|---|
| ChromaDB local + Blob sync | Fast queries (local SSD) + persistence across deployments |
| 1 Gunicorn worker + 4 threads | ChromaDB is not multi-process safe; threads share the same collection object |
| Background thread for ingest | Upload returns immediately; large files don't block the HTTP request |
| Per-file source check (not bulk get) | Avoids SQLite "too many SQL variables" error on large collections |
| SSE streaming for LLM | Perceived latency reduced — words appear instantly instead of 15s wait |
| Min replicas = 1 | Avoids cold start (model loading takes 20-30s from zero) |
| sentences_per_chunk = 3 | Smaller chunks = more focused embeddings = better retrieval accuracy |
| Query expansion (LLM-based) | Handles terminology variants — "mobile" vs "phone", "contact" vs "details" |
| Multi-query retrieval | Run ChromaDB 3 times (original + 2 expansions), deduplicate by ID across results |
| Adjacent chunk expansion | Fetch N-1 and N+1 to provide more context (3 sentences → 9 sentences per chunk) |
| Document-order sorting | Chunks from same source appear sequentially — LLM sees coherent document flow |
| BM25 + RRF hybrid search | IDF-weighted keywords (BM25) + rank fusion robust to score scale differences (k=60) |
| RRF k=60 constant | Industry standard from Cormack et al. (2009) — robust to ties and outliers |
| Per-candidate BM25 corpus | IDF computed over 40-100 retrieved candidates, not full collection — more discriminative |
| Batch delete (500 IDs per batch) | Avoids SQLite "too many SQL variables" error on large file deletion |
| Orphan index cleanup | Per-source batched cleanup endpoint to remove stale index entries |
| Browser TTS (Play button) | Free text-to-speech using Web Speech API — no backend cost |
| Docker Buildx + layer caching | Registry-based build cache for faster CI/CD deployments |
